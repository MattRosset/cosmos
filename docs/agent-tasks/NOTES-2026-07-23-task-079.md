# NOTES — TASK-079 execution (octree flush during flight)

Judgment calls made while implementing TASK-079, logged as they happened (spec-task rule).

## 1. Flush cap value: kept 8 (spec's decision held, but for a re-measured reason)

The spec picked `OCTREE_FLUSH_PER_FRAME_FLYING = 8`. I measured both 8 and 4 live on the
dense pack (M1/Metal, fly-out→fly-in cycle, 3× each): **identical** visibility outcome
(`lumaMax` 765, points drawn rise to ~1.6M) AND identical frame-timing (see #2). Since the
cap did not change either the fix or the cost, I kept the spec value 8 (research rationale:
≥ `maxInFlight` (6) so the drain keeps pace with decode and the queue never backs up on a
machine that decodes faster than this one).

## 2. Discovered: the ~65 ms fly-in frame is PRE-EXISTING, not caused by this change

The spec's manual pass condition said "0 frames > 50 ms" during the fly-in. My first
after-fix runs showed a recurring ~65–70 ms frame (~2 of 3 fly-ins), independent of the
flush cap (8 vs 4 both spiked). That looked like a regression, so I ran a control:
`git stash push apps/web/src/scene/GalaxyScene.tsx`, rebuilt the **unmodified** code, and
re-measured. The original (deferral ON) shows the **same** ~65–68 ms spike on ~2 of 3
fly-ins (fly-in 3 was clean at 22.8 ms). ⇒ the spike is inherent to the scale-jump on this
M1/Metal, NOT introduced by TASK-079.

Consequence: corrected the spec's "Verification beyond the gate" bar from "0 frames > 50 ms"
(which the baseline itself fails) to "no worse than baseline / no frame > 150 ms" (the
breadcrumb-perf `MAX_FRAME_MS`). The fix is a strict improvement: black→visible with no
frame-timing regression.

**Out of scope / follow-up:** the pre-existing intermittent ~65 ms scale-jump hitch is a
separate concern (a real ~4-frame stutter on the breadcrumb transition, deferral-independent).
Not fixed here. Worth a root-cause pass if it bothers anyone — candidate causes: a
GPU-upload/allocation burst or shader state change at the procgen→catalog hand-off. Filed as
a note, not folded into this diff.

## 3. Bug intermittency (why before/after must be measured at matched `outLoaded`)

The black fly-in only reproduces when the fly-out evicted the near field hard
(`streaming.loadedChunks` bottoms ~204). Some fly-outs evict less and the fly-in is already
visible even on the original. So the clean before/after is: at `outLoaded ≈ 204`, original
`lumaMax` stays 15 (black) vs after-fix 765. Runs where the original happened to stay
visible are not counter-evidence — they just didn't hit the precondition.
