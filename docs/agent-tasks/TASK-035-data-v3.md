# Task: `data` v3 — octree manifest + on-demand tile loader

**ID:** TASK-035
**Status:** ✅ DONE
**Target package:** `packages/data`
**Size:** M
**Phase:** 3 — lane (data runtime)
**Depends on:** TASK-031, TASK-032

## Goal

The sanctioned Phase-3 thaw of the `data` public API (additions below only;
TASK-009/TASK-023 behavior is byte-identical and all their tests pass unmodified):
load an `OctreeManifest`, expose the tree structure for traversal, and fetch +
decode individual octree tiles **on demand** into `StarBatch`es (§5.7 "loaded on
demand by `streaming`"). Decoding runs in the worker pool (TASK-032) when a pool is
supplied, on the main thread otherwise (parity, testable in Node). This is the
loader half of the streaming pipeline; `streaming` (TASK-038) drives it.

## Frozen Interface (additions to @cosmos/data — existing API unchanged)

```ts
import type {
  MortonKey, OctreeManifest, OctreeTileManifest, StarBatch,
} from '@cosmos/core-types';
import type { LoadOptions } from './load'; // existing
import type { WorkerPool } from '@cosmos/workers';

/** Thrown on octreeFormatVersion mismatch or schema violation (mirrors PackFormatError). */
export class OctreeFormatError extends Error {}

export interface OctreeNode {
  readonly key: MortonKey;
  readonly manifest: OctreeTileManifest;
  /** Child keys present (ADR-003 childMask), in Morton order. */
  readonly childKeys: readonly MortonKey[];
}

export interface OctreeSource {
  readonly root: OctreeNode;
  readonly context: OctreeManifest['context'];
  readonly rootHalfExtentUnits: number;
  readonly idPrefix: string;
  getNode(key: MortonKey): OctreeNode | undefined;
  /**
   * Fetch (if needed) + decode one tile into a StarBatch. Cancellable via
   * AbortSignal (§5.8 cancel stale requests). When `pool` was supplied to
   * loadOctreePack, decode runs off-thread; else on the main thread. The returned
   * StarBatch's originPc = the node center (galaxy-context parsecs), idPrefix =
   * manifest idPrefix, catalogIds carried from the tile. Rejects with
   * OctreeFormatError on a content-hash/slice mismatch, with an AbortError on abort.
   */
  loadTile(key: MortonKey, opts?: { readonly signal?: AbortSignal }): Promise<StarBatch>;
}

export interface LoadOctreeOptions extends LoadOptions {
  /** When present, tile decode is dispatched to this pool (§5.13). */
  readonly pool?: WorkerPool;
}

/** Fetch + validate the manifest only (NOT the tiles). Rejects wrong
 *  octreeFormatVersion (§11). Tiles are fetched lazily by loadTile. */
export function loadOctreePack(
  manifestUrl: string,
  opts?: LoadOctreeOptions,
): Promise<OctreeSource>;
```

## Fixed semantics (transcribe, don't redesign)

- `loadOctreePack` fetches `octree.json`, validates `octreeFormatVersion` (reject
  mismatch with `OctreeFormatError`, §11) and the manifest shape, and indexes nodes
  by `MortonKey` in a `Map`. It does **not** fetch any `.bin` (§5.7 lazy tiles).
- `loadTile` resolves the tile's `binUrl` relative to the manifest URL, fetches it
  (honoring `opts.signal` — abort cancels both the fetch and any worker dispatch via
  the pool's `CancelToken` wired to the signal), verifies `contentHashSha256` and
  that every `BufferSlice` lies within the `.bin`, then decodes:
  - **With a pool:** `pool.dispatch('octree.decode', { tile, idPrefix, bin },
    { transfer: [bin], token })` — the decode + StarBatch construction happen in the
    worker (the handler lives in this package's worker entry, see Deliverables); the
    resulting StarBatch buffer is transferred back (zero-copy, §5.13).
  - **Without a pool:** decode synchronously on the main thread into a StarBatch
    (same code path the worker handler calls — share the pure decode function).
- A decoded tile's `StarBatch.originPc` = `tile.centerUnits`; positions stay
  tile-local f32 (§5.2 / ADR-003 §3) — the renderer gets the camera-relative offset
  per frame as usual.
- Optional in-memory LRU of decoded tiles is **out of scope here** — caching/eviction
  is `streaming`'s budget responsibility (§5.8). `loadTile` decodes each call;
  callers cache.

## Inputs / Outputs

- **Inputs:** fixture octree pack in `test/fixtures/` (a tiny manifest + 2–3 tiles,
  generated by the TASK-034 fixture builder or hand-written); injected `fetchImpl`.
- **Outputs:** `source.root.childKeys`; `await source.loadTile('0/0')` →
  a `StarBatch` whose `count === tile.pointCount`, `originPc === tile.centerUnits`.

## Constraints & Forbidden Actions

- Do not modify `core-types`. Only the API additions above may change `data`'s public
  surface (this file is the thaw approval). All TASK-009 AND TASK-023 tests pass
  UNMODIFIED — break one ⇒ `blocked`, not the test edited.
- No Three.js, no React, no DOM beyond injectable `fetch` (existing pattern). The
  worker entry file is the ONLY new file that imports `@cosmos/workers`'
  `serveWorker`; the `OctreeSource` runtime imports the `WorkerPool` *type* and is
  handed a pool instance — it does not construct one.
- Allowed dependencies (additions): `@cosmos/workers` (for `WorkerPool` type +
  `serveWorker` in the worker entry). No new third-party deps.
- Tile positions are f32 RELATIVE to the node center — never absolute f32 (§5.2).
- No `Math.random()`. No allocations are required on a frame path here (loadTile is
  async, not per-frame), but the decode itself must not build per-star objects —
  decode straight into the StarBatch typed arrays (mirror `procgen`).

## Common Mistakes (architecture §5.7 — copy kept verbatim)

- Parsing CSV in the browser — n/a; tiles are binary, the manifest is JSON.
- Mixing units — tile positions are galaxy-context parsecs RELATIVE to the node
  center; the renderer applies the camera-relative offset.
- Loading entire Gaia subset eagerly — `loadOctreePack` loads the manifest ONLY;
  tiles load via `loadTile` on demand (do NOT prefetch the whole tree).
- Ignoring missing-data flags — carry `hipIds` (0 = none) through the decode
  unchanged.
- Plus: decoding on the main thread when a pool was supplied (must dispatch);
  forgetting to wire `opts.signal` to the worker `CancelToken` (a fast camera move
  must cancel stale tile loads, §5.8); verifying the content hash AFTER decode
  instead of before (verify the raw `.bin` bytes).

## Acceptance Tests

The task is DONE only when these pass in CI:

1. `pnpm --filter @cosmos/data test` — new `test/octree.test.ts` (fixture-driven,
   injected fetch; both with a fake `WorkerPool` and without):
   - `loadOctreePack`: valid fixture loads; the manifest URL resolves tile `binUrl`s
     relatively; wrong `octreeFormatVersion` throws `OctreeFormatError`; a tile
     whose `BufferSlice` exceeds its `.bin` length throws `OctreeFormatError`.
   - `OctreeSource`: `root`, `getNode`, `childKeys` (Morton order, matches
     `childMask`); unknown key → undefined.
   - `loadTile` (no pool): returns a `StarBatch` with `count === pointCount`,
     `originPc === centerUnits`, `idPrefix` carried, `catalogIds`/`hipIds` carried;
     reconstructed absolute positions match the fixture.
   - `loadTile` (with a fake pool): dispatches `octree.decode` with the `.bin` in
     the transfer list; the returned batch matches the no-pool decode byte-for-byte
     (parity).
   - **Content hash:** a tile whose bytes don't match `contentHashSha256` rejects
     with `OctreeFormatError`.
   - **Abort:** an aborted `signal` rejects `loadTile` and (with a pool) cancels the
     dispatch (the fake pool records the cancellation).
2. **Coverage gate:** unchanged from TASK-009/023 (do not lower thresholds).
3. `pnpm verify` exits 0 (boundary lint: no Three.js/React; only the worker entry
   imports `@cosmos/workers`).
